# 05 — 轻量桌面技术选型对比

> **状态：已采纳。** 产品实现栈为 **Tauri 2 + React + TypeScript + SQLite**。  
> 本文保留为选型决策记录，供后人理解「为什么不选 Electron / 原生 .NET」；不必再按文中分期重新 Spike。

硬约束：冷启动快、内存低、安装包可控、擅长 Windows 文件复制/监视、本地 SQLite。

## 3. 逐项评估

### A. Tauri 2

| 项 | 评价 |
|----|------|
| 包体 | 优：通常数十 MB 级，复用系统 WebView2 |
| 内存 | 优：无自带 Chromium |
| 文件 IO | 优：Rust 侧高效；需设计清晰 command 边界 |
| 拖拽 | 中：Web 拖拽 + Rust 落盘，要小心路径与权限 |
| 开发速度 | 中：前端快，Rust 插件/命令需学习 |
| 风险 | WebView2 运行时依赖（Win10/11 一般已具备）；部分杀软对未知 exe 敏感 |

### B. Electron 精简

| 项 | 评价 |
|----|------|
| 包体 | 差～中：Chromium 体积大，需强力裁剪与延迟加载 |
| 内存 | 差：闲置也偏高，与「极轻」冲突 |
| 文件 IO | 优：Node `fs` / `chokidar` 成熟 |
| 拖拽 | 优：生态熟悉 |
| 开发速度 | 优：若团队偏 TS 全栈 |
| 风险 | 易膨胀为「又一个 VS Code」；需严格依赖预算 |

### C. .NET WinUI / WPF

| 项 | 评价 |
|----|------|
| 包体 | 优～中：单文件发布 + Trim 可很轻 |
| 内存 | 优 |
| 文件 IO | 优：Windows 一等公民；回收站 API 方便 |
| 拖拽 | 优：原生 |
| 开发速度 | 中：Markdown 友好渲染、现代卡片 UI 要更多自建 |
| 风险 | 跨平台若未来需要则绑定 Windows（本产品当前可接受） |

### D. Flutter Desktop

| 项 | 评价 |
|----|------|
| 包体 | 中 |
| 内存 | 中 |
| 文件 IO | 中：常见能力要 plugin |
| UI | 优：定制视觉强 |
| 开发速度 | 中：若团队无 Flutter 经验则慢 |
| 风险 | Markdown/虚拟列表等要选型；与 Agent 工具链（TS）略远 |

## 4. 加权结论（规划推荐）

**首选推荐：Tauri 2 + 前端 React/Vue（二选一，实现期定）+ SQLite（sidecar 或 Rust 侧）。**

理由对齐约束：

1. 最贴「非常轻量化」  
2. UI 仍可用 Web 技术做三栏工作台与 Markdown 渲染  
3. 索引/哈希/监视放 Rust，避免 UI 线程卡顿  
4. 不阻塞产品定义：前端组件与本文档中的数据模型可先用纯 TS 类型对齐 Schema  

**次选：.NET WinUI** — 若实现团队强 .NET / 要深度 Windows Shell（回收站、Jump List），且接受 UI 偏原生控件风格。

**不推荐作默认：Electron** — 除非包体/内存指标被明确放宽。

## 5. 量化门槛（实现立项时验收）

在参考机（Win10 x64，SSD，无其它重载）上，MVP 构建目标：

| 指标 | 目标 |
|------|------|
| 安装包（不含 WebView2 引导） | ≤ 25 MB（Tauri 路线）或给出偏差说明 |
| 冷启动至可交互 Catalog | ≤ 1.5 s（命中本地索引） |
| 闲置工作集 | ≤ 150 MB（Tauri）；Electron 路线若采用则 ≤ 300 MB 且需书面豁免 |
| 扫描 500 个 skill 增量 | 不阻塞 UI；进度可见 |
